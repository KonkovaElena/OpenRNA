# Release And Verification

## Purpose

This document defines the minimal release contract for the standalone OpenRNA repository and the verification path for consumers who want stronger supply-chain evidence than a source checkout alone.

## Release Contract

- release tags use semantic versioning in the form `vX.Y.Z`
- releases are cut from `main`
- GitHub autogenerated release notes are configured by `.github/release.yml`
- tagged builds should produce three operator-visible artifacts from `.github/workflows/supply-chain-provenance.yml`:
  - `openrna-dist-<sha>.tgz`
  - `openrna-runtime-sbom.cdx.json`
  - `openrna-artifacts.sha256`

## Maintainer Flow

1. Merge reviewed work into `main` only after the source pull request is green on `CI Verify`, `Analyze (javascript-typescript)`, and `Dependency Review`.
2. Create and push an annotated tag such as `v0.1.0`.
3. Wait for the tag run of `CI` and `Supply Chain Provenance` to finish successfully.
4. The tagged `Supply Chain Provenance` run creates the GitHub release automatically if it does not already exist.
5. Verify that the generated release notes and attached assets look correct on the release page.

## Operational Lessons

- `Supply Chain Provenance` expects `openrna-runtime-sbom.cdx.json` to be pure UTF-8 JSON. `actions/attest@v4` rejects wrapper text, shell banners, or encoding-damaged output.
- The canonical SBOM file generator is `npm run sbom:cyclonedx:file`, which delegates to `scripts/write-cyclonedx-sbom.mjs` and rewrites the npm output into normalized JSON.
- Do not create the attested SBOM file with shell redirection from `npm run sbom:cyclonedx`, including forms such as `npm run sbom:cyclonedx > file` or `npm run sbom:cyclonedx -- > file`.
- If the `Attest SBOM` step fails with JSON parse errors such as `Unexpected token '>'`, inspect the generated SBOM artifact first. That failure mode usually means the file contains wrapper text rather than raw JSON.

## Consumer Verification

### 1. Verify checksums

On Unix-like systems:

```bash
sha256sum -c openrna-artifacts.sha256
```

On PowerShell:

```powershell
Get-FileHash .\openrna-dist-<sha>.tgz -Algorithm SHA256
Get-FileHash .\openrna-runtime-sbom.cdx.json -Algorithm SHA256
```

Compare the resulting digests with `openrna-artifacts.sha256`.

### 2. Verify build provenance

```bash
gh attestation verify openrna-dist-<sha>.tgz \
  --repo KonkovaElena/OpenRNA \
  --signer-workflow KonkovaElena/OpenRNA/.github/workflows/supply-chain-provenance.yml \
  --source-ref refs/tags/v0.1.0
```

For a machine-readable result:

```bash
gh attestation verify openrna-dist-<sha>.tgz \
  --repo KonkovaElena/OpenRNA \
  --signer-workflow KonkovaElena/OpenRNA/.github/workflows/supply-chain-provenance.yml \
  --source-ref refs/tags/v0.1.0 \
  --format json
```

### 3. Inspect the SBOM

The repository emits a CycloneDX JSON SBOM. Review `openrna-runtime-sbom.cdx.json` directly or regenerate it locally with:

```bash
npm run sbom:cyclonedx:file
```

## Boundaries

- artifact attestations prove provenance for the generated build bundle; they do not prove clinical validity or regulatory qualification
- the SBOM is a dependency inventory, not a security certification
- GitHub release notes are a convenience layer and do not replace reading the diff or the source tag
- hosted release publication still depends on GitHub Actions permissions and repository settings being left enabled
