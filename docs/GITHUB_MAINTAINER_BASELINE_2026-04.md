# GitHub Maintainer Baseline

## Purpose

This document records the GitHub-side controls that should be enabled for the standalone OpenRNA repository but cannot be fully represented by tracked files alone.

## Branch Protection Baseline For `main`

Configure a single protection rule or ruleset for `main` with the following minimum posture:

| Control | Recommended setting |
|---------|---------------------|
| Pull requests | required before merge |
| Required approvals | at least 1 |
| Code owner review | required |
| Stale approvals | dismiss on new commits |
| Conversation resolution | required |
| Required status checks | `CI Verify`, `Analyze (javascript-typescript)`, `Dependency Review` |
| Status check strictness | strict / branch up to date before merge |
| Force pushes | disabled |
| Branch deletion | disabled |
| Apply to admins | enabled |

Recommended additions when the maintainer workflow can support them cleanly:

- linear history
- merge queue for busy repositories
- signed commits if all maintainers and automation paths are already compatible

GitHub documents that required checks should use unique job names to avoid ambiguous status results. This repository therefore uses explicit check names instead of relying on generic job IDs.

## Security Settings Baseline

| Control | April 2026 guidance |
|---------|---------------------|
| Private vulnerability reporting | enable in repository Settings -> Security -> Advanced Security |
| Security notifications | maintainers should watch the repository for `Security alerts` or `All Activity` |
| Secret scanning | public repositories receive baseline secret scanning automatically |
| Secret Protection / push protection | enable if the repository plan exposes it |
| Dependency graph and Dependabot alerts | keep enabled |
| Code scanning | keep the tracked CodeQL workflow enabled |
| Artifact attestations | keep available for public releases and tagged builds |

## Release Baseline

- `.github/release.yml` is the file-based source of truth for GitHub autogenerated release-note categories
- `RELEASE.md` is the public how-to for maintainers and consumers
- releases should be cut from signed or otherwise verified tags on `main`
- provenance and SBOM evidence should come from the tagged `Supply Chain Provenance` workflow run
- the tagged `Supply Chain Provenance` workflow is expected to publish the GitHub release and attach the build bundle, SBOM, and checksums automatically

## Scope Boundary

This document is a repository-operations baseline. It does not claim that the GitHub settings listed here are already enabled on the hosted repository at every moment; it defines the expected maintainer posture.
